1. Arquitectura

CapaTecnología
BackendNode.js + Express 4
Base de datosSQLite (node:sqlite, built-in)
AuthGoogle OAuth 2.0 + sesiones en SQLite
FrontendVanilla JS (sin framework)
ContenedorDocker (node:24-slim)
HostingFly.io + volumen persistente
CI/CDGitHub Actions → flyctl deploy
Principio clave: un solo código para todos los entornos. La diferencia entre local y producción son variables de entorno, no archivos ni ramas distintas.

2. Modelo de ramas

develop → trabajo local y pruebas (localhost:3000) master → producción (auto-despliega a Fly.io)

Nunca trabajes directo en master. Siempre develop → merge → master.

4. Flujo de trabajo diario

Empezar

git checkout develop
git pull origin develop
npm run dev          # localhost:3000 con auto-reload

Hacer cambios

npm test                          # correr tests
git add .
git commit -m "feat: descripción"
git push origin develop           # backup, NO despliega

Publicar a producción

git checkout master
git merge develop
git push origin master            # ← dispara el despliegue automático
git checkout develop              # volver a trabajar

Verificar

gh run list --limit 3             # estado de la Action

5. Qué pasa al pushear master

1. push a master en GitHub 2. GitHub Action se dispara (.github/workflows/fly-deploy.yml) 3. Ejecuta: flyctl deploy --remote-only (usa FLY_API_TOKEN) 4. Fly construye la imagen Docker (remote builder) 5. Sube la imagen y arranca máquina en yyz 6. Monta el volumen ih_data en /data (datos persisten) 7. Inyecta los fly secrets como variables de entorno 8. App live en https://inventario-hogar-alex.fly.dev
El volumen ih_data NO se toca en los deploys → productos, compras, fotos y sesiones sobreviven a cada actualización.

Nunca pongas secrets en fly.toml, Dockerfile ni código. Solo en fly secrets (prod) o .env (local, gitignored).

7. Variables y secrets

Local — .env (gitignored)

GOOGLE_CLIENT_ID=<client-id-dev>
GOOGLE_CLIENT_SECRET=<secret-dev>
CALLBACK_URL=http://localhost:3000/auth/google/callback
SESSION_SECRET=<texto-largo>
PORT=3000

Producción — fly secrets

fly secrets set \
  GOOGLE_CLIENT_ID="<client-id-prod>" \
  GOOGLE_CLIENT_SECRET="<secret-prod>" \
  SESSION_SECRET="<texto-aleatorio>" \
  CALLBACK_URL="https://inventario-hogar-alex.fly.dev/auth/google/callback" \
  --app inventario-hogar-alex

GitHub — para la Action

FLY_API_TOKEN en repo → Settings → Secrets and variables → Actions.

8. Google OAuth (dos clients)

Un client por entorno aísla los secrets.

ClientRedirect URISecret va en
Desarrollohttp://localhost:3000/auth/google/callback.env local
Producciónhttps://inventario-hogar-alex.fly.dev/auth/google/callbackfly secrets
Si el login da redirect_uri_mismatch, falta registrar el redirect URI en el client correcto (Google Cloud Console → Credentials).

10. Respaldo de la base de datos

La DB vive en el volumen. Bajá copias periódicas:

fly sftp get /data/inventario.db ./backup-inventario.db --app inventario-hogar-alex
Un volumen es persistente pero no es un backup. Descargá copias seguido.

12. Checklist de seguridad

  • .env en .gitignore (nunca commitear)
  • Secrets de prod solo en fly secrets
  • FLY_API_TOKEN solo en GitHub Secrets
  • OAuth client separado dev/prod
  • Cookie secure:true en producción (automático por NODE_ENV)
  • Rotar cualquier secret expuesto
  • Backups periódicos de la DB
